===========================================
PAYMENT SYSTEM ARCHITECTURE
===========================================

1. PAYMENT PLANS (Batches)
   - Master payment plan/batch assigned to customers
   - Example: Batch 10001, Total Price: 1000 SSP
   - Contains: Plan Name, Total Amount, Duration (months), Frequency
   - Used for: Defining the overall payment structure for properties
   - One plan can be assigned to multiple customers

2. PAYMENT SCHEDULES (Installments)
   - Child records of Payment Plans
   - Defines all installments/payments due over X months
   - Example: Installment #1, Amount: 100 SSP, Due Date: Jan 15, 2025
   - Contains: Installment Number, Due Date, Amount, Surcharge Rate
   - Used for: Tracking what payments are due and when
   - System can automatically check if payments are overdue

3. PAYMENTS (Actual Receipts)
   - Records of actual payments received from customers
   - Child of Payment Schedule (links to specific installments)
   - Example: Payment for Installment #1, Amount: 100 SSP, Date: Jan 10, 2025
   - Contains: Payment Date, Amount, Method, Reference Number, Status
   - Used for: Recording actual money received
   - Multiple payments can be made against one schedule/installment

Flow: Payment Plan (Batch) → Payment Schedules (Installments) → Payments (Receipts)





===========================================
CRITICAL BUSINESS RULES: CUSTOMER & PROPERTY RELATIONSHIP
===========================================

** 1 PROPERTY = 1 CUSTOMER (OR 0 IF NOT ALLOTTED) **
** 1 CUSTOMER = 0 OR 1 PROPERTY (NEVER MULTIPLE) **

Rules:
1. A Property can have:
   - NO customer (Status: Available - not allotted)
   - ONE customer (Status: Allotted)
   - NEVER multiple customers/allotments

2. A Customer (Membership) can have:
   - NO property (when new customer, installments pending)
   - ONE property (after allotment)
   - NEVER multiple properties

3. If a customer wants ANOTHER plot:
   - Create a NEW Customer entry for the same person
   - Use a NEW RegistrationID
   - Assign a NEW Payment Plan
   - Reason: One Registration = One Payment Plan = One Property

4. Database Enforcement:
   - UNIQUE constraint on Allotment.PropertyID
   - Each property can only have ONE record in Allotment table
   - System will reject duplicate allotment attempts

5. Allotment Display:
   - Property Details → Allotment Tab: Show ONLY 1 allotment (or none)
   - Never show multiple allotments for same property

===========================================
REGISTRATION MODULE WORKFLOW
===========================================

Registration is the FIRST STEP in the customer onboarding process.

Process Flow:
1. Registration Created (Status: Pending)
   - Captures basic info: Name, CNIC, Phone, Email
   - Admin reviews the registration
   
2. Registration Reviewed
   - Admin changes status to "Approved" or "Rejected"
   - If Approved → Can proceed to create Customer
   
3. Customer Creation
   - Links to Registration (RegID)
   - Assigns Payment Plan
   - Full customer profile created
   - Registration → Customer (conversion)

Business Rules:
- One Registration can have multiple Customers (if same person buys multiple plots)
- Each Customer must link to one Registration
- Registration cannot be deleted if linked to customer(s)
- Registration approval is recommended before customer creation
